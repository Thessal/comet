# Sequence Notation

The Comet compiler implements a bidirectional converter between its internal execution graph (`DagBuilder` representation) and a flat sequence of tokens. This sequence is designed to be easily fed into and generated by a Transformer model for search algorithms.

The format relies on a **Polish Prefix Notation** traversal (DFS pre-order) to implicitly represent the tree structure of mathematical expressions. The `DagBuilder`'s hash-consing ensures that variable assignments and duplicate operations are automatically resolved when reconstructing the graph.

## Token Structure

The sequence consists of a linear list of string tokens representing nodes in the AST graph.

### 1. Function Calls (`CALL`)
A function or behavior call node is represented by a `CALL` token, followed by its arguments.

**Format:** `CALL|<func_name>|<arity>`
- `<func_name>`: The name of the function, behavior, or operator (e.g., `Diff`, `Compare`, `ts_mean`).
- `<arity>`: An integer representing exactly how many argument branches will immediately follow this token.

### 2. Argument Boundaries (`ARG`)
Because Comet supports both structural (positional) and named arguments, every argument branch explicitly announces itself before the child sequence starts.

**Format:** `ARG|<arg_name>` or `ARG|`
- `<arg_name>`: The explicitly bound name of the argument (e.g., `ARG|signal`).
- If no name is provided (positional), it simply yields `ARG|`.

*Directly following an `ARG` token is the full prefix sequence of the child expression.*

### 3. Literals and Identifiers
Leaf nodes in the expression tree are encoded natively as strings. During reconstruction from the sequence, these are disambiguated by parsing rules:

- **Strings:** Enclosed in literal double quotes (e.g., `"volume"`).
- **Integers:** Parsed mathematically (e.g., `21`, `-5`).
- **Floats:** Parsed mathematically (e.g., `0.1`, `3.14`).
- **Booleans:** Exact literal `true` or `false`.
- **Identifiers:** Any token that does not match the above rules is evaluated as a runtime identifier or variable name (e.g., `data`, `volume_signal`).

## Examples

### Simple Call
Suppose we have a simple expression: `A(arg1=B, arg2=C)`.

The resulting Polish sequence is:
```text
[
  "CALL|A|2",
  "ARG|arg1", "B",
  "ARG|arg2", "C"
]
```

### Nested Call
Suppose we have a nested expression: `Compare(signal="volume", reference=Historical(days=21))`

The resulting Polish sequence is:
```text
[
  "CALL|Compare|2",
  "ARG|signal", "\"volume\"",        // Child 1 (String Literal)
  "ARG|reference",                   // Child 2 (Function Call)
    "CALL|Historical|1",
    "ARG|days", "21"
]
```

## Hash-Consing and Assignments

The Sequence does not need to have explicit `Assign` operations. When translating the `DagBuilder` graph to a Sequence, assignments expand out fully due to the depth-first traversal of the expression tree roots.

When reconstructing the DAG back from the sequence, Comet uses its native `DagBuilder` which has **Hash-Consing**. 
If a Transformer generates:
```text
[
  "CALL|Add|2",
  "ARG|", "X",
  "ARG|", "X"
]
```
The rebuilder will intercept that `"X"` has identical hash identity, preventing duplicate computation mapping and efficiently creating a Directed Acyclic Graph instead of a naive tree layout.
